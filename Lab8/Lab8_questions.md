## РАЗДЕЛ 1: АННОТАЦИИ В JAVA

---

**Вопрос 1: Что такое аннотация в Java? Зачем она нужна?**

Аннотация — это форма метаданных (данных о данных), которую можно добавлять к классам, методам, полям, параметрам и другим элементам Java-кода. Аннотации сами по себе не изменяют логику выполнения программы — они служат "метками" или "подсказками" для:
1. Компилятора — например, @Override говорит компилятору проверить, действительно ли метод переопределяет родительский
2. Инструментов разработки — IDE, анализаторы кода
3. Фреймворков во время выполнения — Spring, JUnit, Hibernate читают аннотации через Reflection и выполняют действия
В нашей лабораторной @DataProcessor помечает методы, которые DataManager должен найти и выполнить.

---

**Вопрос 2: Что такое @Retention и какие у неё значения? Почему мы использовали RUNTIME?**

@Retention определяет, как долго аннотация "живёт" в коде:
- `RetentionPolicy.SOURCE` — аннотация существует только в исходном коде (.java), при компиляции удаляется. Пример: @SuppressWarnings
- `RetentionPolicy.CLASS` — аннотация сохраняется в .class файле, но недоступна во время выполнения программы. Это поведение по умолчанию.
- `RetentionPolicy.RUNTIME` — аннотация сохраняется в .class файле И доступна во время выполнения через Reflection.

Мы использовали RUNTIME, потому что DataManager читает аннотацию @DataProcessor во время работы программы через метод `method.isAnnotationPresent(DataProcessor.class)`. Если бы мы не указали RUNTIME, метод вернул бы false и наши обработчики никогда не вызвались бы.

---

**Вопрос 3: Что такое @Target и для чего он нужен?**

@Target ограничивает места, где можно применять аннотацию. Значения ElementType:
- `METHOD` — только к методам
- `TYPE` — к классам, интерфейсам, перечислениям
- `FIELD` — к полям класса
- `PARAMETER` — к параметрам метода
- `CONSTRUCTOR` — к конструкторам
- `LOCAL_VARIABLE` — к локальным переменным

Мы использовали `ElementType.METHOD`, потому что @DataProcessor должна отмечать только методы обработки данных. Если студент попробует поставить её на класс или поле — компилятор выдаст ошибку. Это защита от неправильного использования.

---

**Вопрос 4: Как объявляется пользовательская аннотация? Чем @interface отличается от interface?**

Пользовательская аннотация объявляется с ключевым словом `@interface` (а не просто `interface`):
```java
public @interface MyAnnotation {
    String value() default "hello";
}
```
Отличие от обычного интерфейса:
- `@interface` — специальный синтаксис только для аннотаций
- Элементы аннотации объявляются как методы без тела (не как поля)
- Элементы могут иметь значения по умолчанию (`default`)
- Аннотацию нельзя инстанциировать напрямую (`new MyAnnotation()` — ошибка)
- Аннотация неявно расширяет java.lang.annotation.Annotation

---

**Вопрос 5: Что такое элементы аннотации? Почему они выглядят как методы?**

Элементы аннотации объявляются как методы без тела:
```java
public @interface DataProcessor {
    String description() default "";
    int priority() default 1;
}
```
Они "выглядят как методы" потому что технически аннотация в Java — это специальный вид интерфейса, а элементы аннотации — абстрактные методы этого интерфейса. Когда мы пишем `@DataProcessor(description = "Фильтр")`, мы фактически заполняем эти поля. Прочитать значение можно через `annotation.description()`.

`default ""` — задаёт значение по умолчанию, чтобы не нужно было указывать элемент при каждом использовании аннотации.

---

**Вопрос 6: Какие встроенные аннотации Java вы знаете?**

Встроенные аннотации в Java:
- `@Override` — сообщает компилятору, что метод переопределяет метод суперкласса. Если метод не найден в родителе — ошибка компиляции.
- `@Deprecated` — помечает элемент как устаревший. IDE предупреждает при его использовании.
- `@SuppressWarnings("тип")` — отключает конкретные предупреждения компилятора.
- `@FunctionalInterface` — проверяет, что интерфейс имеет ровно один абстрактный метод (функциональный интерфейс).
- `@SafeVarargs` — подавляет предупреждения о безопасности дженериков с varargs.

## РАЗДЕЛ 2: REFLECTION (РЕФЛЕКСИЯ)

---

**Вопрос 7: Что такое Reflection в Java? Где он используется?**

Reflection — механизм Java, позволяющий программе во время выполнения (runtime) исследовать и изменять свою структуру: получать информацию о классах, методах, полях, конструкторах, аннотациях.

Что можно делать через Reflection:
- Получить все методы класса: `clazz.getDeclaredMethods()`
- Создать объект класса по имени: `Class.forName("MyClass").newInstance()`
- Вызвать метод по объекту Method: `method.invoke(объект, аргументы)`
- Прочитать и изменить приватные поля

Где используется:
- **JUnit** — находит методы с @Test и вызывает их
- **Spring** — сканирует аннотации @Component, @Autowired и настраивает бины
- **Hibernate** — читает @Entity, @Column и маппит объекты на таблицы БД
- **Jackson** — сериализует/десериализует объекты в JSON

В нашей лабораторной мы используем Reflection, чтобы найти методы с @DataProcessor и вызвать их без явного обращения по имени метода.

---

**Вопрос 8: Что делает метод getDeclaredMethods()? Чем он отличается от getMethods()?**

- `getDeclaredMethods()` — возвращает все методы, объявленные непосредственно в данном классе (public, protected, package-private, private). НЕ включает унаследованные методы.
- `getMethods()` — возвращает только public методы данного класса И всех его суперклассов и интерфейсов (включая унаследованные).

Мы используем `getDeclaredMethods()`, потому что нас интересуют только методы, объявленные в конкретном классе-обработчике.

---

**Вопрос 9: Как работает method.invoke()? Что произойдёт, если метод приватный?**

`method.invoke(объект, аргументы)` вызывает метод у указанного объекта с переданными аргументами:
```java
List<String> result = (List<String>) method.invoke(processor, data);
```
- `processor` — объект, у которого вызываем метод
- `data` — аргумент, передаваемый методу
- Возвращает Object, который нужно привести к нужному типу

Если метод приватный, сначала нужно вызвать `method.setAccessible(true)` — иначе будет выброшено `IllegalAccessException`. Это даёт доступ к приватным методам через Reflection (что является обходом инкапсуляции и должно использоваться осторожно).

---

**Вопрос 10: Какие недостатки у использования Reflection?**

1. **Производительность** — Reflection намного медленнее прямого вызова методов, т.к. требует динамического разрешения имён во время выполнения.
2. **Безопасность** — можно обойти модификаторы доступа (private, protected).
3. **Сложность отладки** — ошибки проявляются во время выполнения, а не при компиляции.
4. **Хрупкость** — рефакторинг (переименование метода) может сломать Reflection-код без ошибки компиляции.
5. **Ограничения оптимизатора** — JIT-компилятор хуже оптимизирует Reflection-вызовы.

## РАЗДЕЛ 3: STREAM API

---

**Вопрос 11: Что такое Stream API? Когда он был добавлен в Java?**

Stream API добавлен в Java 8 (2014 год). Это API для обработки последовательностей элементов (коллекций, массивов, файлов) в функциональном стиле.

Stream — это не коллекция и не хранилище данных. Это "конвейер" (pipeline) для обработки данных:
```
источник данных → промежуточные операции → терминальная операция → результат
```
Пример: `list.stream().filter(...).map(...).collect(toList())`

Stream ленивый (lazy): промежуточные операции не выполняются немедленно — они выполняются только когда вызвана терминальная операция. Это позволяет оптимизировать обработку.

---

**Вопрос 12: Чем отличается промежуточная операция от терминальной?**

**Промежуточные (Intermediate)** — возвращают новый Stream, могут быть цепочкой:
- `filter(Predicate)` — оставляет элементы, удовлетворяющие условию
- `map(Function)` — преобразует каждый элемент
- `flatMap(Function)` — "разворачивает" вложенные коллекции
- `distinct()` — убирает дубликаты
- `sorted()` — сортирует
- `limit(n)` — берёт первые n элементов
- `skip(n)` — пропускает первые n элементов
- `peek(Consumer)` — действие над каждым элементом без изменения потока (для отладки)

**Терминальные (Terminal)** — завершают цепочку, возвращают результат (не Stream):
- `collect(Collector)` — собирает в коллекцию
- `count()` — количество элементов
- `forEach(Consumer)` — выполняет действие для каждого
- `findFirst()` — первый элемент (Optional)
- `anyMatch(Predicate)` — есть ли хоть один подходящий элемент
- `reduce(identity, accumulator)` — свёртка (например, сумма)
- `min()`, `max()` — минимум/максимум

---

**Вопрос 13: Что такое filter()? Как работает лямбда-выражение внутри него?**

`filter(Predicate<T> predicate)` — промежуточная операция, оставляющая только элементы, для которых предикат возвращает `true`.

Лямбда `line -> line.length() >= 5`:
- `line` — параметр (текущий элемент потока, имя произвольное)
- `->` — разделитель параметров и тела
- `line.length() >= 5` — тело лямбды, возвращает boolean

Stream проходит по каждому элементу, вычисляет предикат. Если `true` — элемент остаётся в потоке, если `false` — удаляется.

---

**Вопрос 14: Что такое map()? В чём разница между map() и filter()?**

- `filter()` — отбирает элементы (убирает часть). Тип элементов не меняется.
- `map(Function<T, R>)` — преобразует каждый элемент. Тип может измениться.

Пример `map`:
```java
.map(String::toUpperCase)    // Stream<String> → Stream<String>
.map(s -> s.length())        // Stream<String> → Stream<Integer>
.map(n -> n * 2)             // Stream<Integer> → Stream<Integer>
```

В нашем коде `map(String::trim)` — преобразует каждую строку, обрезая пробелы по краям.

---

**Вопрос 15: Что такое collect()? Какие варианты Collectors существуют?**

`collect(Collector)` — терминальная операция, "собирающая" поток в результирующую структуру.

Часто используемые Collectors:
- `Collectors.toList()` — в List
- `Collectors.toSet()` — в Set (без дубликатов, неупорядоченно)
- `Collectors.toMap(keyFn, valueFn)` — в Map
- `Collectors.joining(", ")` — конкатенация строк с разделителем
- `Collectors.groupingBy(fn)` — группировка в Map<K, List<V>>
- `Collectors.counting()` — подсчёт количества
- `Collectors.toUnmodifiableList()` — в немодифицируемый List

В нашем коде мы всегда используем `Collectors.toList()`.

---

**Вопрос 16: Что такое distinct() и sorted()? Как они работают?**

`distinct()` — использует метод `equals()` для сравнения элементов. Оставляет только уникальные элементы, убирая все повторения. Для корректной работы тип элементов должен правильно переопределять equals() и hashCode().

`sorted()` — сортирует элементы в "естественном порядке". Для String — лексикографически (по коду символов): сначала цифры, потом заглавные буквы A-Z, потом строчные a-z.
Есть перегрузка `sorted(Comparator)` для пользовательского порядка:
```java
.sorted(Comparator.reverseOrder())  // обратный порядок
.sorted((a, b) -> a.length() - b.length())  // по длине строки
```

---

**Вопрос 17: Что такое method reference (ссылка на метод)? Как она связана с лямбдами?**

Ссылка на метод — сокращённая запись лямбды, которая просто вызывает существующий метод. Синтаксис: `Класс::метод` или `объект::метод`.

Виды ссылок на методы:
1. Статический метод: `Integer::parseInt` = `s -> Integer.parseInt(s)`
2. Метод экземпляра (для типа): `String::toUpperCase` = `s -> s.toUpperCase()`
3. Метод конкретного экземпляра: `System.out::println` = `s -> System.out.println(s)`
4. Конструктор: `ArrayList::new` = `() -> new ArrayList<>()`

В нашем коде:
- `String::trim` вместо `line -> line.trim()`
- `String::toUpperCase` вместо `line -> line.toUpperCase()`

---

**Вопрос 18: Что значит "Stream ленивый"? Приведите пример.**

"Ленивый" (lazy) значит, что промежуточные операции не выполняются немедленно — они регистрируются как "план действий" и выполняются только при вызове терминальной операции.

Пример: предположим, у нас список из 1000 элементов:
```java
list.stream()
    .filter(s -> s.startsWith("A"))  // ещё не выполняется
    .map(String::toUpperCase)         // ещё не выполняется
    .findFirst();                     // ЗАПУСКАЕТ выполнение
```
При вызове `findFirst()` Stream не обработает все 1000 элементов. Он найдёт первый элемент, начинающийся с "A", применит toUpperCase и вернёт результат. Остальные элементы не будут обработаны.

Это называется "short-circuit evaluation" (вычисление с прерыванием) — оптимизация производительности.

## РАЗДЕЛ 4: java.util.concurrent — МНОГОПОТОЧНОСТЬ

---

**Вопрос 19: Что такое ExecutorService? Зачем он нужен вместо прямого создания потоков?**

`ExecutorService` — интерфейс в `java.util.concurrent` для управления пулом потоков.

Раньше потоки создавали напрямую:
```java
Thread t = new Thread(() -> { ... });
t.start();
```
Проблемы прямого создания потоков:
- Создание потока — дорогостоящая операция (выделение памяти под стек)
- Если создавать потоки на каждую задачу — производительность падает
- Тяжело контролировать количество одновременных потоков

ExecutorService решает эти проблемы:
- Создаёт пул потоков заранее
- Переиспользует потоки для разных задач
- Управляет очередью задач
- Позволяет легко получить результат через Future

---

**Вопрос 20: Чем отличается newFixedThreadPool() от других методов Executors?**

Фабричные методы класса Executors:

- `newFixedThreadPool(n)` — создаёт пул из ровно n потоков. Все задачи распределяются между ними. Если все потоки заняты — новая задача ждёт в очереди. Хорошо когда заранее известно количество задач.

- `newCachedThreadPool()` — пул с неограниченным числом потоков. Если все заняты — создаётся новый поток. Неиспользуемые потоки уничтожаются через 60 секунд. Хорошо для задач с непредсказуемой нагрузкой.

- `newSingleThreadExecutor()` — пул из одного потока. Задачи выполняются последовательно. Гарантирует, что в данный момент выполняется только одна задача.

- `newScheduledThreadPool(n)` — позволяет запускать задачи по расписанию или с задержкой.

В нашем коде мы используем `newFixedThreadPool(processors.size())` — создаём столько потоков, сколько у нас обработчиков.

---

**Вопрос 21: Что такое Future? Как работает future.get()?**

`Future<V>` — интерфейс, представляющий "будущий результат" асинхронной задачи.

Когда мы вызываем `executor.submit(задача)`, задача не выполняется в главном потоке немедленно. Метод возвращает Future — "обещание" того, что результат будет готов позже.

Методы Future:
- `get()` — блокирует текущий поток и ждёт результата. Возвращает результат или бросает исключение.
- `get(timeout, unit)` — ждёт не более указанного времени. Если не успело — `TimeoutException`.
- `isDone()` — возвращает true если задача завершена (успешно или с ошибкой).
- `isCancelled()` — завершена ли задача отменой.
- `cancel(true)` — пытается отменить задачу.

В нашем коде `future.get()` в главном потоке гарантирует, что мы дождёмся завершения каждого потока перед тем, как перейти к saveData().

---

**Вопрос 22: Что такое CopyOnWriteArrayList? Почему нельзя использовать обычный ArrayList в многопоточном коде?**

Проблема с ArrayList: если два потока одновременно добавляют элементы в один ArrayList, может произойти:
- `ConcurrentModificationException` — исключение при одновременной модификации
- Потеря данных — один поток перезаписывает изменения другого
- Corrupted state — внутренний массив ArrayList приходит в некорректное состояние

`CopyOnWriteArrayList` из пакета `java.util.concurrent` решает эту проблему:
- При каждой операции записи (add, set, remove) создаётся **новая копия** внутреннего массива
- Операции чтения (get, iterator) работают со старой копией без блокировок
- Итерация всегда безопасна — итератор работает со снимком списка на момент создания

Недостатки: операции записи медленнее (копирование массива), занимает больше памяти. Подходит когда читают часто, а пишут редко.

Альтернативы: `Collections.synchronizedList()`, `Vector` (устаревший), `BlockingQueue`.

---

**Вопрос 23: Что такое executor.shutdown()? Зачем его нужно вызывать?**

`executor.shutdown()` — завершает работу пула потоков:
1. Не принимает больше новых задач
2. Дожидается завершения уже запущенных и ожидающих в очереди задач
3. Освобождает ресурсы (память, системные потоки)

Если не вызвать `shutdown()`:
- Программа может не завершиться (потоки из пула работают как non-daemon и не дают JVM выйти)
- Утечка ресурсов — потоки остаются в памяти

Связанные методы:
- `shutdownNow()` — пытается немедленно остановить все потоки (посылает interrupt)
- `awaitTermination(timeout, unit)` — блокирует текущий поток, ожидая завершения пула с таймаутом
- `isShutdown()` — был ли вызван shutdown
- `isTerminated()` — завершены ли все задачи после shutdown

---

**Вопрос 24: Что произойдёт, если не вызвать future.get()?**

Если не вызвать `future.get()`, главный поток не будет ждать завершения задач. Возможные последствия:
1. Метод `saveData()` может вызваться раньше, чем `processData()` заполнит список результатов — файл будет пустым или неполным.
2. Исключения внутри потоков будут проигнорированы — мы не узнаем об ошибках.

`future.get()` — это барьер синхронизации, гарантирующий, что мы продолжаем выполнение только после завершения всех потоков.

## РАЗДЕЛ 5: ООП И СТРУКТУРА ПРОГРАММЫ

---

**Вопрос 25: Зачем мы разделили обработчики на отдельные классы (в обычном варианте)?**

Разделение на классы FilterProcessor, TransformProcessor, AggregateProcessor следует принципу **Single Responsibility Principle (SRP)** — принципу единственной ответственности из SOLID:
- Каждый класс отвечает только за одну задачу
- Классы легко тестировать независимо
- Новый обработчик добавляется без изменения существующих классов (Open/Closed Principle)
- Код более читаемый и понятный

В простом варианте все методы в одном классе DataProcessors — это проще, но менее масштабируемо.

---

**Вопрос 26: Зачем список processors объявлен как List<Object>, а не List<FilterProcessor>?**

`List<Object>` позволяет хранить объекты любых классов. Если бы мы написали `List<FilterProcessor>`, то могли бы регистрировать только объекты FilterProcessor, а не TransformProcessor или другие.

Используя тип Object (родитель всех классов Java), метод `registerDataProcessor(Object processor)` принимает любой объект. DataManager не знает заранее, какой тип обработчика будет зарегистрирован — он находит нужные методы через Reflection в runtime. Это делает систему гибкой и расширяемой.

---

**Вопрос 27: Что такое try-with-resources? Почему мы используем его для файлов?**

`try (ресурс)` — конструкция, гарантирующая автоматическое закрытие ресурса после завершения блока, даже при возникновении исключения:
```java
try (BufferedReader reader = new BufferedReader(new FileReader(source))) {
    // используем reader
} // reader.close() вызывается автоматически здесь
```
Без try-with-resources:
```java
BufferedReader reader = null;
try {
    reader = new BufferedReader(new FileReader(source));
    // используем reader
} finally {
    if (reader != null) reader.close(); // нужно закрывать вручную!
}
```
Работает с классами, реализующими интерфейс `AutoCloseable` (или `Closeable`). Все потоки ввода-вывода Java реализуют этот интерфейс.

---

**Вопрос 28: Что такое BufferedReader и зачем он нужен?**

`FileReader` — читает файл посимвольно. Каждый вызов read() обращается к диску — медленно.

`BufferedReader` — оборачивает FileReader и добавляет буфер. Читает большой блок данных за один раз, сохраняет в памяти (буфере), и отдаёт данные по запросу из буфера — быстро.

Дополнительно BufferedReader предоставляет метод `readLine()` для построчного чтения и метод `lines()` возвращающий `Stream<String>`.

Аналогично: `FileWriter` + `BufferedWriter` — буферизированная запись. `write()` пишет в буфер, и данные сбрасываются на диск разом (или при вызове `flush()`/`close()`).

---

**Вопрос 29: Зачем в методах объявлено throws IOException / throws Exception?**

`throws` в сигнатуре метода объявляет "проверяемые исключения" (checked exceptions), которые метод может выбросить. Вызывающий код обязан либо:
1. Обработать исключение блоком `try-catch`
2. Объявить его в своей сигнатуре `throws`

`IOException` — исключение при работе с вводом-выводом (файл не найден, нет прав доступа и т.д.). Все методы работы с файлами могут его выбросить.

`Exception` — базовый класс всех исключений. Используем в `processData()`, потому что там могут возникнуть разные типы исключений: `InvocationTargetException` (от Reflection), `ExecutionException` (от Future.get()), `InterruptedException` (при прерывании ожидания).

## РАЗДЕЛ 6: ОБЩИЕ ВОПРОСЫ ПО ТЕМЕ

---

**Вопрос 30: Что такое многопоточность? Почему она полезна?**

Многопоточность — выполнение нескольких задач одновременно в рамках одной программы. Каждая задача выполняется в отдельном потоке (Thread).

Преимущества:
- **Производительность** — на многоядерных процессорах задачи выполняются действительно параллельно
- **Отзывчивость** — UI-поток не блокируется долгими операциями
- **Использование пауз** — пока один поток ждёт ввода-вывода, другой выполняет вычисления

В нашей лабораторной каждый обработчик данных работает в своём потоке — фильтрация, трансформация и агрегация выполняются параллельно.

---

**Вопрос 31: Что такое race condition (состояние гонки)? Как мы её избегаем?**

Race condition — ситуация, когда результат программы зависит от порядка выполнения потоков. Пример:
```
Поток 1: читает processedData.size() = 5
Поток 2: читает processedData.size() = 5
Поток 1: добавляет 3 элемента → size = 8
Поток 2: добавляет 2 элемента → перезаписывает часть данных!
```
Мы избегаем race condition, используя `CopyOnWriteArrayList` вместо обычного ArrayList. При каждой операции записи `addAll()` создаётся новая копия внутреннего массива — операция атомарна и другие потоки не могут "вмешаться" в середине.

---

**Вопрос 32: Что такое лямбда-выражение? Что такое функциональный интерфейс?**

**Лямбда-выражение** — анонимная функция, которую можно передавать как аргумент.
Синтаксис: `параметры -> тело`
```java
(int a, int b) -> a + b        // два параметра, возвращает сумму
s -> s.toUpperCase()           // один параметр, возвращает строку
() -> System.out.println("Hi") // нет параметров, нет возвращаемого значения
```

**Функциональный интерфейс** — интерфейс с ровно одним абстрактным методом. Лямбды реализуют функциональные интерфейсы.

Основные функциональные интерфейсы из java.util.function:
- `Predicate<T>` — `boolean test(T t)` — условие (для filter)
- `Function<T, R>` — `R apply(T t)` — преобразование (для map)
- `Consumer<T>` — `void accept(T t)` — действие (для forEach)
- `Supplier<T>` — `T get()` — поставщик значения
- `Runnable` — `void run()` — задача без параметров и результата (для потоков)

---

**Вопрос 33: Как работает конвейер Stream API в нашем коде? Опишите шаг за шагом.**

Рассмотрим на примере FilterProcessor:
```java
data.stream()
    .filter(line -> !line.isBlank())
    .filter(line -> line.length() >= 5)
    .collect(Collectors.toList());
```

Шаги:
1. `data.stream()` — создаём Stream из списка. Ничего ещё не выполняется — только создаётся конвейер.
2. `.filter(line -> !line.isBlank())` — регистрируем первую промежуточную операцию. Всё ещё ничего не выполняется.
3. `.filter(line -> line.length() >= 5)` — регистрируем вторую промежуточную операцию.
4. `.collect(Collectors.toList())` — ТЕРМИНАЛЬНАЯ операция. Именно здесь запускается весь конвейер!

Stream берёт первый элемент из data, пропускает его через filter1, затем через filter2. Если прошёл оба — добавляет в результат. Затем берёт второй элемент и т.д.

---

**Вопрос 34: Что такое submit() у ExecutorService? Чем он отличается от execute()?**

`execute(Runnable)` — запускает задачу, не возвращает результат, не сообщает об исключениях.

`submit(Runnable)` — запускает задачу, возвращает `Future<?>`. Через Future можно:
- Дождаться завершения (`future.get()`)
- Получить исключение, если задача завершилась ошибкой

`submit(Callable<V>)` — то же, но задача возвращает значение. Future<V>.get() возвращает это значение.

В нашем коде мы используем `submit()`, чтобы получить Future и дождаться завершения всех потоков через `future.get()`.

---

**Вопрос 35: Почему порядок элементов в output.txt может отличаться при каждом запуске?**

Потому что у нас параллельная обработка — три потока работают одновременно и добавляют результаты в `processedData` в том порядке, в котором успевают завершить работу. Порядок завершения потоков непредсказуем и зависит от:
- Загрузки процессора
- Планировщика операционной системы
- Объёма данных, обрабатываемых каждым потоком

Это нормально для многопоточных программ. Если порядок важен — нужно явно синхронизировать или объединять результаты после завершения всех потоков.

**Вопрос 36: Как можно расширить программу (добавить новый обработчик)?**

В обычном варианте достаточно:
1. Создать новый класс, например `StatisticsProcessor`
2. Написать метод с нужной логикой и пометить его `@DataProcessor`
3. В Main добавить одну строку: `manager.registerDataProcessor(new StatisticsProcessor())`

Больше ничего менять не нужно! DataManager автоматически найдёт новые методы через Reflection. Это демонстрирует принцип **Open/Closed** — открыто для расширения, закрыто для модификации.

---

**Вопрос 37: Что такое @SuppressWarnings("unchecked") и почему он стоит в нашем коде?**

Компилятор Java проверяет типы во время компиляции. Когда мы пишем:
```java
List<String> result = (List<String>) method.invoke(processor, data);
```
Компилятор не может проверить во время компиляции, действительно ли `method.invoke()` вернёт `List<String>` (это станет известно только во время выполнения через Reflection). Компилятор выдаёт предупреждение "unchecked cast" (непроверяемое приведение типов).

`@SuppressWarnings("unchecked")` подавляет это предупреждение. Это означает: "Я, программист, уверен что приведение типов корректно — не предупреждай меня об этом". Если приведение некорректно — во время выполнения возникнет ClassCastException.

---

**Вопрос 38: Что такое пакет java.util.concurrent? Какие классы из него вы использовали?**

`java.util.concurrent` — пакет Java для работы с многопоточностью (добавлен в Java 5, существенно расширен в Java 8).

Содержит:
- **Исполнители**: ExecutorService, Executors, ScheduledExecutorService
- **Синхронизаторы**: CountDownLatch, CyclicBarrier, Semaphore, Phaser
- **Атомарные типы**: AtomicInteger, AtomicBoolean, AtomicReference
- **Потокобезопасные коллекции**: CopyOnWriteArrayList, ConcurrentHashMap, BlockingQueue
- **Future/CompletableFuture**: для асинхронного программирования
- **Блокировки**: ReentrantLock, ReadWriteLock

В нашей лабораторной мы использовали:
- `ExecutorService` — управление пулом потоков
- `Executors` — фабрика для создания ExecutorService
- `Future<?>` — результат асинхронной задачи
- `CopyOnWriteArrayList<String>` — потокобезопасный список

---

**Вопрос 39: Можно ли было реализовать задание без многопоточности? Зачем тогда она нужна?**

Да, технически можно вызвать все методы последовательно без потоков. Но в задании явно требуется использование `java.util.concurrent` для эффективной параллельной обработки.

Преимущество многопоточности в нашей задаче:
- Если обработчиков много и каждый долго работает — параллельная обработка значительно быстрее
- На 4-ядерном процессоре: 4 обработчика, работая параллельно, завершатся примерно в 4 раза быстрее, чем последовательно
- Это особенно важно при обработке больших файлов или сложных трансформациях

**Вопрос 40: Что произойдёт, если input.txt не существует?**

При вызове `new FileReader("input.txt")` будет выброшено исключение `FileNotFoundException` (наследник IOException). Поскольку метод `loadData` объявляет `throws IOException`, исключение передастся в метод `main`. В main объявлено `throws Exception`, поэтому исключение передастся в JVM, программа аварийно завершится с сообщением об ошибке:
```
Exception in thread "main" java.io.FileNotFoundException: input.txt (No such file or directory)
```

Для продакшн-кода нужно добавить проверку:
```java
File file = new File(source);
if (!file.exists()) {
    throw new IllegalArgumentException("Файл не найден: " + source);
}
```
